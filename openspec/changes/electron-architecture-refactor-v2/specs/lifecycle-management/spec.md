## 新增需求

### 需求: 服务注册
Application 类必须允许服务在启动前显式声明依赖关系并注册。

#### 场景: 注册无依赖的服务
- **当** 服务使用空依赖数组注册时
- **则** 系统必须将服务存储在注册表中，无阻塞依赖

#### 场景: 注册有依赖的服务
- **当** 服务使用依赖 ID ["env", "config"] 注册时
- **则** 系统必须将服务存储在注册表中，并标记为被阻塞直到依赖启动

#### 场景: 拒绝重复注册
- **当** 相同 ID 的服务被注册两次时
- **则** 系统必须抛出错误，指示重复注册

#### 场景: 启动后拒绝注册
- **当** Application.start() 调用后尝试注册服务时
- **则** 系统必须抛出错误，指示仅允许在初始化阶段注册

### 需求: 拓扑排序
Application 类必须使用拓扑排序（Kahn 算法）计算尊重所有服务依赖关系的有效启动顺序。

#### 场景: 排序线性依赖链
- **当** 服务注册为 A→B→C（线性链）时
- **则** 计算的顺序必须是 [A, B, C]

#### 场景: 排序并行依赖
- **当** 服务注册为 A、B（都不依赖任何东西）、C（依赖 A 和 B）时
- **则** 计算的顺序必须以 A 和 B（任意顺序）开始，然后是 C

#### 场景: 检测循环依赖
- **当** 服务注册时存在循环依赖（A→B→A）时
- **则** 系统必须在 start() 期间抛出错误，指示循环依赖

#### 场景: 排序复杂依赖图
- **当** 服务具有菱形依赖模式（A→B, A→C, B→D, C→D）时
- **则** 计算的顺序必须确保 A 在 B 和 C 之前启动，D 在 B 和 C 之后启动

### 需求: 顺序服务启动
Application 类必须按拓扑顺序顺序启动服务，等待每个服务完成后再启动下一个。

#### 场景: 成功启动所有服务
- **当** 所有服务启动时没有错误
- **则** Application 状态转换为 'ready'，所有服务状态为 'started'

#### 场景: 处理必需服务失败
- **当** 必需服务（required=true）在 start() 期间失败时
- **则** 系统必须按相反顺序停止所有已启动的服务并抛出错误

#### 场景: 跳过可选服务失败
- **当** 可选服务（required=false）在 start() 期间失败时
- **则** 系统必须记录警告，将服务标记为 'failed'，并继续启动剩余服务

#### 场景: 失败时自动回滚
- **当** 服务 C 在启动序列 [A, B, C, D] 中失败时
- **则** 系统必须按相反顺序停止 B 和 A，然后抛出原始错误

### 需求: 服务检索
Application 类必须通过 get<T>(id: string) 方法提供类型安全的已启动服务访问。

#### 场景: 检索已启动的服务
- **当** gateway 服务启动后调用 get("gateway") 时
- **则** 系统必须返回 GatewayService 实例

#### 场景: 拒绝检索未启动的服务
- **当** gateway 服务启动前调用 get("gateway") 时
- **则** 系统必须抛出错误，指示服务不可用

#### 场景: 编译时类型安全
- **当** 在 TypeScript 中使用 const gateway = app.get<GatewayService>("gateway") 时
- **则** 编译器必须推断正确类型并提供自动完成

### 需求: 优雅关闭
Application 类在调用 stop() 时必须按反向拓扑顺序停止所有服务。

#### 场景: 干净停止所有服务
- **当** 服务顺序为 [A, B, C] 时调用 stop()
- **则** 系统必须按相反顺序 [C, B, A] 停止服务，并转换到 'stopped' 状态

#### 场景: 出错时继续关闭
- **当** 服务 B 在 stop() 期间抛出错误时
- **则** 系统必须记录错误并继续停止剩余服务（A）

#### 场景: 关闭期间阻止操作
- **当** Application 状态为 'stopping' 时调用 get() 或 start()
- **则** 系统必须抛出错误，指示关闭期间不允许操作

### 需求: 状态机强制
Application 类必须强制执行有效的状态转换: initializing → starting → ready → stopping → stopped。

#### 场景: 强制状态转换顺序
- **当** 状态为 'initializing' 时调用 stop()
- **则** 系统必须抛出错误，指示无效的状态转换

#### 场景: 停止后允许重启
- **当** Application 停止后调用 start()
- **则** 系统必须允许重新初始化并再次启动所有服务

#### 场景: 幂等停止
- **当** stop() 被多次调用时
- **则** 只有第一次调用必须执行关闭逻辑，后续调用必须立即返回

### 需求: 依赖验证
Application 类必须验证所有声明的服务依赖都引用已注册的服务。

#### 场景: 拒绝缺失依赖
- **当** 服务 B 注册时依赖 "nonexistent" 时
- **则** 系统必须在 start() 期间抛出错误，指示依赖不存在

#### 场景: 注册时验证依赖
- **当** 服务注册时带有依赖项
- **则** 系统必须验证所有依赖 ID 存在于注册表中，或标记为在启动时验证

## 新增需求

### 需求: 单元测试框架
项目必须使用 Vitest 进行单元测试，目标行覆盖率为 80%。

#### 场景: 运行单元测试
- **当** 执行 npm run test:unit 时
- **则** Vitest 必须运行 tests/unit/**/*.test.ts 中的所有测试

#### 场景: 覆盖率阈值
- **当** 覆盖率低于 80% 行、85% 函数或 70% 分支时
- **则** npm run test:coverage 必须以非零代码退出

#### 场景: 快速执行
- **当** 执行 npm run test:unit 时
- **则** 所有单元测试必须在 10 秒内完成

#### 场景: 监视模式
- **当** 执行 npm run test:watch 时
- **则** Vitest 必须在文件变更时重新运行受影响的测试

### 需求: E2E 测试框架
项目必须使用 Playwright 进行端到端测试，覆盖关键用户流程。

#### 场景: 运行 E2E 测试
- **当** 执行 npm run test:e2e 时
- **则** Playwright 必须启动 Electron 应用并运行 tests/e2e/**/*.spec.ts 中的测试

#### 场景: 无头执行
- **当** E2E 测试在 CI 环境中运行时
- **则** Electron 必须在无头模式下运行，不显示窗口

#### 场景: 失败时截图
- **当** E2E 测试失败时
- **则** Playwright 必须捕获失败状态的截图

#### 场景: 测试超时
- **当** E2E 测试超过 30 秒时
- **则** 测试必须以超时错误失败

### 需求: 测试组织
测试必须按类型（单元、集成、E2E）组织，并具有清晰的命名约定。

#### 场景: 单元测试结构
- **当** 为 src/services/env.service.ts 编写单元测试时
- **则** 测试文件必须位于 tests/unit/services/env.service.test.ts

#### 场景: 集成测试结构
- **当** 为 IPC 通信编写集成测试时
- **则** 测试文件必须位于 tests/integration/ipc-communication.test.ts

#### 场景: E2E 测试结构
- **当** 为启动流程编写 E2E 测试时
- **则** 测试文件必须位于 tests/e2e/startup.spec.ts

### 需求: Mock 工具
测试框架必须提供用于 mock 服务、IPC 和外部依赖的工具。

#### 场景: Mock Service 接口
- **当** 测试 Application 类时
- **则** 测试必须能够创建实现 Service 接口的 mock 服务

#### 场景: Mock IPC renderer
- **当** 测试 Main 进程 IPC 处理器时
- **则** 测试必须能够模拟 ipcRenderer.invoke() 调用

#### 场景: Mock 子进程
- **当** 测试 ProcessManager 时
- **则** 测试必须能够 mock spawn() 而不创建真实进程

### 需求: CI 集成
测试套件必须在所有分支和拉取请求的 CI 中自动运行。

#### 场景: Push 时运行测试
- **当** 代码被推送到 main 或 develop 分支时
- **则** GitHub Actions 必须运行所有测试套件（单元、集成、E2E）

#### 场景: 失败时阻止 PR
- **当** 拉取请求中任何测试失败时
- **则** PR 必须被阻止合并直到测试通过

#### 场景: 覆盖率报告
- **当** 测试在 CI 中完成时
- **则** 覆盖率报告必须上传到 Codecov

#### 场景: 快速反馈
- **当** 测试在 CI 中运行时
- **则** 单元测试必须在 E2E 测试之前运行（快速失败）

### 需求: 测试隔离
测试必须是独立的，测试用例之间没有共享的可变状态。

#### 场景: 测试间重置 mock
- **当** 测试使用 vi.fn() 来 mock 函数时
- **则** mock 必须在下一个测试前自动重置

#### 场景: 独立测试顺序
- **当** 测试被执行时
- **则** 测试执行顺序不得影响通过/失败结果

#### 场景: 清洁数据库状态
- **当** 集成测试使用数据库时
- **则** 每个测试必须从干净状态开始（没有前一个测试的数据）

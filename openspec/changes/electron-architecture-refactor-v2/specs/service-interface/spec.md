## 新增需求

### 需求: 标准接口契约
所有应用服务必须实现 Service 接口，包含 id、required 标志和生命周期方法（start、stop、isHealthy）。

#### 场景: 实现必需属性
- **当** 服务类实现 Service 接口时
- **则** 服务必须定义只读的 id（string）和 required（boolean）属性

#### 场景: 实现 start 方法
- **当** 服务实现 Service.start() 时
- **则** 方法必须是异步的并返回 Promise<void>

#### 场景: 实现 stop 方法
- **当** 服务实现 Service.stop() 时
- **则** 方法必须是异步的并返回 Promise<void>

#### 场景: 实现 isHealthy 方法
- **当** 服务实现 Service.isHealthy() 时
- **则** 方法必须是同步的并返回 boolean

### 需求: 服务生命周期状态
服务必须维护显式状态转换: pending → starting → started → stopping → stopped，failed 作为错误状态。

#### 场景: 初始状态为 pending
- **当** 服务在 Application 中注册时
- **则** 服务状态必须是 'pending'

#### 场景: 转换到 starting
- **当** Application 调用 service.start() 时
- **则** 服务状态必须转换为 'starting'

#### 场景: 成功时转换到 started
- **当** service.start() 完成且无错误时
- **则** 服务状态必须转换为 'started'

#### 场景: 出错时转换到 failed
- **当** service.start() 抛出错误时
- **则** 服务状态必须转换为 'failed' 并存储错误

#### 场景: 防止重复启动
- **当** 在 'started' 状态的服务上调用 service.start() 时
- **则** 系统必须抛出错误或立即返回（幂等）

### 需求: 健康检查报告
服务必须通过 isHealthy() 提供实时健康状态，反映操作就绪情况。

#### 场景: 运行正常时报告健康
- **当** 在所有子系统正常运行的服务上调用 isHealthy() 时
- **则** 方法必须返回 true

#### 场景: 子系统失败时报告不健康
- **当** 在关键子系统失败的服务上调用 isHealthy() 时
- **则** 方法必须返回 false

#### 场景: 快速执行要求
- **当** 调用 isHealthy() 时
- **则** 方法必须在 100ms 内完成（不进行网络调用或重计算）

### 需求: 可选错误回调
服务可以实现 onError(error: Error) 回调进行自定义错误处理。

#### 场景: 失败时调用错误回调
- **当** 服务实现 onError() 且 start() 期间发生错误时
- **则** Application 必须在转换到 'failed' 状态前调用 onError() 并传入错误

#### 场景: 未实现时跳过回调
- **当** 服务未实现 onError() 且发生错误时
- **则** Application 必须继续默认错误处理而不抛出异常

### 需求: 可选指标收集
服务可以实现 getMetrics() 以公开操作指标供监控。

#### 场景: 返回指标对象
- **当** 在实现它的服务上调用 getMetrics() 时
- **则** 方法必须返回包含指标名称-值对的 Record<string, any>

#### 场景: 包含标准指标
- **当** 服务实现 getMetrics() 时
- **则** 返回的对象应该包含运行时间、错误计数或其他相关操作指标

#### 场景: 未实现时跳过指标
- **当** 在未实现 getMetrics() 的服务上调用时
- **则** Application 必须优雅地处理 undefined 返回

### 需求: 幂等生命周期方法
服务的 start() 和 stop() 方法必须是幂等的 - 多次调用安全且无副作用。

#### 场景: 幂等启动
- **当** start() 在同一服务实例上被调用两次时
- **则** 第二次调用必须立即返回或抛出清晰错误，但不重新初始化资源

#### 场景: 幂等停止
- **当** stop() 在同一服务实例上被调用两次时
- **则** 第二次调用必须立即返回，不尝试再次停止

#### 场景: 启动前停止
- **当** 在从未启动过的服务上调用 stop() 时
- **则** 方法必须立即返回且无错误

### 需求: 服务标识
每个服务必须有用于依赖解析和检索的唯一字符串标识符。

#### 场景: 唯一 ID 强制
- **当** 两个服务使用相同 id 注册时
- **则** Application 必须在第二次注册时抛出错误

#### 场景: ID 格式验证
- **当** 服务使用无效 id（空字符串、仅空格）注册时
- **则** Application 必须抛出错误

#### 场景: ID 用于检索
- **当** 调用 app.get("gateway") 时
- **则** 系统必须返回以 id="gateway" 注册的服务

### 需求: 必需与可选服务
服务必须声明它们对应用运行是必需的（required=true）还是可选的（required=false）。

#### 场景: 必需服务失败时中止
- **当** 必需服务在 start() 期间失败时
- **则** Application 必须中止启动并回滚

#### 场景: 可选服务失败时继续
- **当** 可选服务在 start() 期间失败时
- **则** Application 必须记录警告并继续启动剩余服务

#### 场景: 服务必需属性不可变
- **当** 访问服务的 required 属性时
- **则** 它必须是只读的，注册后不可修改

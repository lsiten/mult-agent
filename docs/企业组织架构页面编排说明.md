# 企业组织架构页面编排说明

> 基于文档：
> - `docs/Hermes Agent 企业组织架构管理平台.md`
> - `docs/hermes-profile-usage-guide.md`
> - Hermes 官方文档：[配置文件：运行多个 Agent](https://hermesagent.org.cn/docs/user-guide/profiles)
>
> 文档类型：页面说明文档  
> 目标版本：V1.0  
> 适用对象：产品 / 设计 / 前端 / 后端

---

## 1. 文档目标

本文档用于定义 Hermes 企业组织架构页面的编排方案。该页面聚焦于一个核心目标：

**让用户在一个页面内完成公司创建、公司目标定义、部门目标定义、岗位配置，以及岗位下 Agent 的编排与管理。**

该页面不是全量企业后台，也不是单纯的组织树展示页，而是一个面向多 Agent 团队的组织配置与编排中心。

---

## 2. 设计来源与约束

本页面方案综合两份现有文档的核心思想：

### 2.1 来自企业组织架构平台文档的约束

- 组织结构以 `公司 -> 部门 -> 岗位 -> Agent` 为核心管理路径
- 企业管理关注的是组织关系、公司目标、部门目标和工作协作
- 每个 Agent 需要独立隔离
- 公司空间和个人空间必须严格区分
- 管理员和部门负责人只能管理企业协作部分，不能访问个人隐私空间

### 2.2 来自 Hermes Profile 使用指南的约束

- 每个 Agent 创建后，系统应基于该 Agent 自动生成独立的 Hermes Profile Agent
- Profile 提供配置、记忆、技能、会话、Gateway 和环境的隔离能力
- Profile 是页面中 Agent 实例的底层运行载体
- 当前 Profile 完整能力主要建立在 CLI 模式
- Electron 当前不直接支持启动时选择 Profile，因此页面设计应先聚焦“组织编排层”，底层实例隔离可通过现有 Profile 机制承载

---

## 3. 页面定位

该页面定位为：

**企业多 Agent 组织架构配置页**

页面主要解决四件事：

1. 创建公司并定义公司一句话核心目标
2. 在公司下创建多个部门，并定义每个部门的一句话目标
3. 在部门下创建岗位，并定义岗位职责
4. 在岗位下编排和管理 Agent，并为每个 Agent 自动生成对应的 Profile Agent 实例

一句话描述：

> 用户先搭建公司的组织结构，再定义各层目标和岗位职责，最后把 Agent 放到具体岗位中执行对应职责。

## 4. 页面范围

### 4.1 V1 页面内必须覆盖的内容

- 公司创建
- 公司一句话目标设置
- 部门创建
- 部门一句话目标设置
- 部门下岗位的新增、编辑和排序
- 岗位下 Agent 的新增、编辑、启停和归属调整
- Agent 创建后生成的 Hermes Profile Agent 信息展示
- 企业空间可管理范围展示

### 4.2 V1 页面暂不覆盖的内容

- 全量任务中心
- 辞退归档完整流程页
- 批量导入导出
- 审计报表
- 个人空间操作页面

V1 应优先保证“组织结构清晰、目标清晰、Agent 编排清晰”。

## 5. 核心信息架构

页面采用四层核心对象模型：

### 5.1 公司

公司是页面的顶层对象，负责定义整个组织的总目标与组织容器。

关键属性：

- 公司名称
- 公司一句话核心目标
- 公司描述
- 公司状态
- 公司工作目录
- 部门数量
- 岗位数量
- Agent 总数

### 5.2 部门

部门是公司目标的承接单元，用于拆分业务职责，并承载多个岗位。

关键属性：

- 部门名称
- 部门一句话目标
- 部门职责说明
- 部门负责人
- 部门工作目录
- 岗位数量
- 部门状态

### 5.3 岗位

岗位是部门目标向执行层落地的责任单元，用于定义一类工作职责，并承载多个 Agent。

关键属性：

- 岗位名称
- 所属部门
- 岗位职责
- 岗位目标
- 编制人数 / Agent 数量
- 默认模板或 Profile 生成策略
- 岗位工作目录
- 岗位状态

### 5.4 Agent

Agent 是人的概念，表示组织中的具体员工个体。每个 Agent 归属于某个岗位，创建后系统基于该 Agent 自动生成独立的 Profile Agent。

关键属性：

- Agent 名称
- 所属部门
- 所属岗位
- Agent 身份 / 角色
- 服务目标
- Agent 工作目录
- Profile Agent 生成状态
- 启用状态
- 公司空间状态
- 个人空间状态说明

## 6. 页面主流程

该页面建议按以下操作路径组织：

### 6.1 第一步：创建公司

用户首次进入页面时，先创建公司。

输入项：

- 公司名称
- 公司一句话核心目标
- 公司描述（可选）

目标：

- 建立组织管理的根节点
- 明确企业级总方向

### 6.2 第二步：创建部门

公司创建完成后，用户可以为公司添加多个部门。

每个部门输入项：

- 部门名称
- 部门一句话目标
- 部门职责说明（可选）
- 部门负责人（可选）

目标：

- 将公司的总目标拆解为多个部门目标
- 让后续 Agent 编排有明确归属

### 6.3 第三步：在部门下创建岗位

每个部门都可以创建多个岗位。

每个岗位输入项：

- 岗位名称
- 所属部门
- 岗位职责
- 岗位目标
- 默认能力要求或模板（可选）
- 编制人数（可选）

目标：

- 将部门目标进一步拆解为可执行职责单元
- 为 Agent 的分配和管理建立稳定结构

### 6.4 第四步：在岗位下编排 Agent

每个岗位都可以创建多个 Agent。

每个 Agent 输入项：

- Agent 名称
- 所属部门
- 所属岗位
- Agent 身份 / 角色
- 负责的子目标
- Profile Agent 生成策略或模板（可选）
- 是否启用

目标：

- 让每个 Agent 都归属于具体岗位
- 让 Agent 的职责与岗位职责、部门目标对应

### 6.5 第五步：统一查看组织结构

完成配置后，页面应展示：

- 公司总目标
- 部门目标列表
- 每个部门下的岗位结构
- 每个岗位下的 Agent 编排结构
- 每个 Agent 的运行与归属信息

## 9. 页面中的编排逻辑

本页面中的“编排”不是单纯添加多个 Agent，而是将 Agent 作为“人”的个体，放入可管理的业务结构中。

### 9.1 编排主线

编排关系应遵循：

`公司目标 -> 部门目标 -> 岗位职责 -> Agent 执行`

这条链路应在页面上保持清晰可读。

### 9.2 编排规则

- 一个公司可以有多个部门
- 一个部门可以有多个岗位
- 一个岗位可以有多个 Agent
- 一个 Agent 在 V1 中仅归属一个岗位
- 一个部门目标可以由多个岗位协同承接
- 一个岗位目标可以由多个 Agent 协同完成
- 每个 Agent 必须有明确职责说明

### 9.3 编排结果

用户在页面中最终看到的是：

- 顶层有一个公司总目标
- 中层有多个部门目标
- 中下层有多个岗位职责
- 底层有多个被编排到岗位中的 Agent

## 10. 与 Hermes Profile 的映射关系

页面中的 Agent 组织对象，需要在创建后生成对应的 Hermes Profile Agent 实例。

### 10.1 映射原则

- 每个 Agent 创建后自动生成一个独立 Profile Agent
- Profile Agent 遵循 Hermes Profile 的独立目录模型，具备独立的 `HERMES_HOME`
- Profile 提供独立的配置、记忆、技能、会话和运行环境
- 页面负责管理组织关系与人员信息
- Profile 负责承载实例隔离

### 10.2 页面应展示的 Profile 信息

建议在 Agent 详情中展示：

- 生成的 Profile Agent 名称
- Profile Agent 状态
- Profile Agent 模板来源或生成策略
- 最近同步时间

### 10.3 当前实现边界

由于当前 Hermes Profile 完整能力主要在 CLI 模式下成熟，页面设计应先以“组织管理层”为主，底层实例控制可逐步接入。

页面文案中不应让用户误以为当前 Electron 已完整支持任意 Profile 切换，而应强调：

- 页面管理的是组织架构和 Agent 编排
- Agent 的底层隔离依赖 Hermes Profile

### 10.4 参考 Hermes Profile 的实现约束

根据 Hermes Profile 文档，Profile 的本质是一个独立目录，内部包含独立的：

- `config.yaml`
- `.env`
- `SOUL.md`
- memories
- sessions
- skills
- Gateway 运行状态与相关文件

因此本页面不应把 Profile 仅理解为一个配置 ID，而应将其视为：

- 一个由系统自动创建的独立 Agent 运行环境
- 一个从组织架构页面派生出来的运行时实例
- 一个与 Agent 人员实体一一对应的隔离工作单元

## 11. 权限与隐私边界

该页面必须遵循企业组织架构文档中的双空间原则。

### 11.1 页面可管理范围

页面可管理内容：

- 公司结构
- 部门结构
- 岗位组织归属
- Agent 组织归属
- Agent 企业角色
- 公司空间相关配置

### 11.2 页面不可触达范围

页面不可提供：

- 个人空间浏览入口
- 个人对话内容查看
- 个人记忆查看
- 个人私有文件访问
- 个人空间导出入口

### 11.3 页面中的说明要求

当用户查看 Agent 详情时，应明确展示：

- 公司空间：企业协作内容，可由企业管理
- 个人空间：仅 Agent 本人可见，本页面不可访问

该提示既是权限说明，也是产品信任边界。

## 13. 存储设计总原则

本页面涉及的数据应拆分为三类存储：

1. SQLite 中的组织主数据
2. 文件系统中的工作目录与运行目录
3. 基于数据库字段渲染生成的 `SOUL.md` / Profile Agent 配置

整体原则如下：

- 组织关系、权限状态、页面展示所需字段，必须以 SQLite 为唯一真源
- 公司、部门、岗位、Agent 各自拥有独立工作目录，工作目录属于文件系统资源
- Agent 的人格、工作风格、行为约束可以以结构化字段存库，再渲染为 `SOUL.md`
- Profile Agent 是基于 Agent 自动生成的派生运行对象，不是组织主数据源
- 不允许将组织关系仅写入 `SOUL.md` 而不落库

推荐在 Hermes 现有 `state.db` 之外，单独维护一个组织数据库：

- Electron 根目录下的 `state.db`：Hermes 现有会话与运行时状态
- `org/org.db`：组织架构、人物实体、岗位、目录、Profile Agent 元数据

这样可以避免业务主数据和 Hermes 运行时数据耦合。

在 Electron 模式下，推荐采用以下标准布局：

```text
~/Library/Application Support/hermes-agent-electron/
├── state.db
├── config.yaml
├── SOUL.md
├── org/
│   ├── org.db
│   ├── migrations/
│   ├── workspaces/
│   ├── profiles/
│   └── rendered/
```

其中：

- `state.db` 保持为 Hermes 现有主运行时数据库
- `org/org.db` 专门存储组织编排相关数据
- `org/` 下的其余目录用于承载组织模块的迁移、目录、Profile Agent 和渲染结果

## 14. SQLite 数据模型设计

### 14.1 建议表清单

V1 建议至少包含以下数据表：

- `companies`
- `departments`
- `positions`
- `agents`
- `agent_personas`
- `profile_agents`
- `workspaces`

### 14.2 companies

存储公司主信息。

建议字段：

- `id`：自增主键
- `code`：公司稳定编码
- `name`：公司名称
- `goal`：公司一句话目标
- `description`：公司描述
- `status`：状态
- `workspace_path`：公司工作目录
- `created_at`
- `updated_at`

### 14.3 departments

存储部门主信息。

建议字段：

- `id`
- `company_id`
- `parent_id`：可选，支持子部门

- `name`
- `goal`
- `description`
- `leader_agent_id`
- `workspace_path`
- `sort_order`
- `status`
- `created_at`
- `updated_at`

### 14.4 positions

存储岗位定义。

建议字段：

- `id`
- `department_id`
- `code`
- `name`
- `goal`
- `responsibilities`
- `headcount`
- `template_key`
- `workspace_path`
- `sort_order`
- `status`
- `created_at`
- `updated_at`

### 14.5 agents

存储 Agent 人员实体。这里的 Agent 是“人”，不是抽象运行实例。

建议字段：

- `id`

## 16. Agent Persona、SOUL 与 Profile Agent 生成规则

### 16.1 生成链路

创建 Agent 后，系统应按以下顺序处理：

1. 写入 `agents`
2. 写入或初始化 `agent_personas`
3. 创建 Agent 工作目录
4. 渲染 `SOUL.md`
5. 创建 `profile_agents`
6. 初始化 Profile Agent 目录
7. 回写 `profile_status`

### 16.2 什么内容写入 Persona / SOUL

建议写入 `agent_personas` 并渲染到 `SOUL.md` 的内容：

- Agent 名称
- 公司名称
- 部门名称
- 岗位名称
- 公司目标
- 部门目标
- 岗位目标
- 岗位职责
- 工作风格
- 输出偏好
- 协作规则
- 禁止事项

### 16.3 什么内容不写入 SOUL 作为真源

以下字段即使在 `SOUL.md` 中出现，也只能是说明性文本，不能作为系统真源：

- 数据库主键和外键
- 组织归属关系
- 上级关系
- 头像路径
- 状态字段
- 权限字段
- 工作目录路径
- Profile Agent 路径

### 16.4 SOUL 模板结构建议

{{position_responsibilities}}

{{working_style}}

{{output_preferences}}

## 20. 页面示例场景

### 14.1 示例

公司：

- 名称：Hermes AI Lab
- 核心目标：构建一个高效协作的企业级多 Agent 团队

部门：

- 市场部
  - 目标：负责市场洞察、内容策划和品牌传播
- 研发部
  - 目标：负责产品开发、平台稳定性和能力演进

岗位：

- 市场部
  - 市场分析岗
  - 内容策划岗
- 研发部
  - 功能交付岗
  - 测试评审岗

Agent 编排：

- 市场分析岗
  - Market Analyst Agent
- 内容策划岗
  - Content Planner Agent
- 功能交付岗
  - Feature Delivery Agent
- 测试评审岗
  - QA Review Agent

这个示例体现的是：

- 公司有一个总目标
- 每个部门承接一类子目标
- 每个岗位承接一类明确职责
- 每个 Agent 作为人，被编排到具体岗位中执行职责

## 21. V1 页面验收标准

V1 页面完成后，至少应满足以下要求：

1. 用户可以创建一个公司，并填写一句话核心目标
2. 用户可以在公司下创建多个部门，并为每个部门填写一句话目标
3. 用户可以在部门下新增、编辑和管理多个岗位
4. 用户可以在岗位下新增、编辑和管理多个 Agent
5. 用户可以清晰查看 `公司 -> 部门 -> 岗位 -> Agent` 的结构关系
6. 用户可以看到 Agent 自动生成的 Profile Agent 信息
7. 页面不提供个人空间访问入口
8. 页面整体逻辑体现“目标驱动的组织编排”

## 23. 总结

本页面的核心不是做一个静态组织树，而是建立一个企业多 Agent 的编排入口。

页面的核心逻辑应始终保持为：

**先定义公司目标，再拆分部门目标，再建立岗位职责，最后把 Agent 编排进岗位中执行目标。**

在这个模型中：

- 公司负责方向
- 部门负责承接目标
- 岗位负责沉淀职责
- Agent 作为人负责执行职责
- Hermes Profile 负责底层隔离

这就是该页面在产品层和架构层上的统一表达。

# 企业组织架构页面编排说明

> 基于文档：
> - `docs/Hermes Agent 企业组织架构管理平台.md`
> - `docs/hermes-profile-usage-guide.md`
> - Hermes 官方文档：[配置文件：运行多个 Agent](https://hermesagent.org.cn/docs/user-guide/profiles)
>
> 文档类型：页面说明文档  
> 目标版本：V1.0  
> 适用对象：产品 / 设计 / 前端 / 后端

---

## 1. 文档目标

本文档用于定义 Hermes 企业组织架构页面的编排方案。该页面聚焦于一个核心目标：

**让用户在一个页面内完成公司创建、公司目标定义、部门目标定义、岗位配置，以及岗位下 Agent 的编排与管理。**

该页面不是全量企业后台，也不是单纯的组织树展示页，而是一个面向多 Agent 团队的组织配置与编排中心。

---

## 2. 设计来源与约束

本页面方案综合两份现有文档的核心思想：

### 2.1 来自企业组织架构平台文档的约束

- 组织结构以 `公司 -> 部门 -> 岗位 -> Agent` 为核心管理路径
- 企业管理关注的是组织关系、公司目标、部门目标和工作协作
- 每个 Agent 需要独立隔离
- 公司空间和个人空间必须严格区分
- 管理员和部门负责人只能管理企业协作部分，不能访问个人隐私空间

### 2.2 来自 Hermes Profile 使用指南的约束

- 每个 Agent 创建后，系统应基于该 Agent 自动生成独立的 Hermes Profile Agent
- Profile 提供配置、记忆、技能、会话、Gateway 和环境的隔离能力
- Profile 是页面中 Agent 实例的底层运行载体
- Profile 的底层实现仍复用 CLI 模式的成熟能力
- Electron 已通过转接层承接这套 Profile 能力，因此页面可以直接编排 Agent 并驱动底层 Profile Agent 创建与管理
- 主 Agent 指默认新建对话时使用的默认 Agent
- 子 Agent 指各个独立的 Profile Agent
- 子 Agent 在创建 Profile 时可以参考主 Agent 的公开数据与工具，但运行时只依赖自己的 Profile Agent

---

## 3. 页面定位

该页面定位为：

**企业多 Agent 组织架构配置页**

页面主要解决四件事：

1. 创建公司并定义公司一句话核心目标
2. 在公司下创建多个部门，并定义每个部门的一句话目标
3. 在部门下创建岗位，并定义岗位职责
4. 在岗位下编排和管理 Agent，并为每个 Agent 自动生成对应的 Profile Agent 实例

一句话描述：

> 用户先搭建公司的组织结构，再定义各层目标和岗位职责，最后把 Agent 放到具体岗位中执行对应职责。

## 4. 页面范围

### 4.1 V1 页面内必须覆盖的内容

- 公司创建

- 部门创建

- 部门下岗位的新增、编辑和排序

## 5. 核心信息架构

页面采用四层核心对象模型：

### 5.1 公司

公司是页面的顶层对象，负责定义整个组织的总目标与组织容器。

关键属性：

- 公司名称
- 公司一句话核心目标
- 公司描述
- 公司状态
- 公司工作目录
- 部门数量
- 岗位数量
- Agent 总数

### 5.2 部门

部门是公司目标的承接单元，用于拆分业务职责，并承载多个岗位。

关键属性：

- 部门名称
- 部门一句话目标
- 部门职责说明
- 部门负责人
- 部门工作目录
- 岗位数量
- 部门状态

### 5.3 岗位

岗位是部门目标向执行层落地的责任单元，用于定义一类工作职责，并承载多个 Agent。

关键属性：

- 岗位名称
- 所属部门
- 岗位职责
- 岗位目标
- 编制人数 / Agent 数量
- 默认模板或 Profile 生成策略
- 岗位工作目录
- 岗位状态

### 5.4 Agent

Agent 是人的概念，表示组织中的具体员工个体。在本系统中，默认新建对话使用的是主 Agent，而组织架构页面中创建和管理的均为子 Agent。每个 Agent 归属于某个岗位，创建后系统基于该 Agent 自动生成独立的 Profile Agent。

关键属性：

- Agent 名称
- 所属部门
- 所属岗位
- Agent 身份 / 角色
- 服务目标
- Agent 工作目录
- Agent 类型（主 Agent / 子 Agent）
- Profile Agent 生成状态
- 启用状态
- 公司空间状态
- 个人空间状态说明

## 6. 页面主流程

该页面建议按以下操作路径组织：

### 6.1 第一步：创建公司

用户首次进入页面时，先创建公司。

输入项：

- 公司名称
- 公司一句话核心目标
- 公司描述（可选）

目标：

- 建立组织管理的根节点
- 明确企业级总方向

### 6.2 第二步：创建部门

公司创建完成后，用户可以为公司添加多个部门。

每个部门输入项：

- 部门名称
- 部门一句话目标
- 部门职责说明（可选）
- 部门负责人（可选）

目标：

- 将公司的总目标拆解为多个部门目标
- 让后续 Agent 编排有明确归属

### 6.3 第三步：在部门下创建岗位

每个部门都可以创建多个岗位。

每个岗位输入项：

- 岗位名称
- 所属部门
- 岗位职责
- 岗位目标
- 默认能力要求或模板（可选）
- 编制人数（可选）

目标：

- 将部门目标进一步拆解为可执行职责单元
- 为 Agent 的分配和管理建立稳定结构

### 6.4 第四步：在岗位下编排 Agent

每个岗位都可以创建多个 Agent。

每个 Agent 输入项：

- Agent 名称
- 所属部门
- 所属岗位
- Agent 身份 / 角色
- 负责的子目标
- Profile Agent 生成策略或模板（可选）
- 是否启用

目标：

- 让每个 Agent 都归属于具体岗位
- 让 Agent 的职责与岗位职责、部门目标对应

### 6.5 第五步：统一查看组织结构

完成配置后，页面应展示：

- 公司总目标
- 部门目标列表
- 每个部门下的岗位结构
- 每个岗位下的 Agent 编排结构
- 每个 Agent 的运行与归属信息

## 9. 页面中的编排逻辑

本页面中的“编排”不是单纯添加多个 Agent，而是将 Agent 作为“人”的个体，放入可管理的业务结构中。

### 9.1 编排主线

编排关系应遵循：

`公司目标 -> 部门目标 -> 岗位职责 -> Agent 执行`

这条链路应在页面上保持清晰可读。

### 9.2 编排规则

- 一个公司可以有多个部门
- 一个部门可以有多个岗位
- 一个岗位可以有多个 Agent
- 主 Agent 是默认新对话使用的默认 Agent
- 公司架构页面中新建的 Agent 默认均为子 Agent / Profile Agent
- 一个 Agent 在 V1 中仅归属一个岗位
- 一个部门目标可以由多个岗位协同承接
- 一个岗位目标可以由多个 Agent 协同完成
- 每个 Agent 必须有明确职责说明

### 9.3 编排结果

用户在页面中最终看到的是：

- 顶层有一个公司总目标
- 中层有多个部门目标
- 中下层有多个岗位职责
- 底层有多个被编排到岗位中的 Agent

## 10. 与 Hermes Profile 的映射关系

页面中的 Agent 组织对象，需要在创建后生成对应的 Hermes Profile Agent 实例。

### 10.1 映射原则

- 每个 Agent 创建后自动生成一个独立 Profile Agent
- Profile Agent 遵循 Hermes Profile 的独立目录模型，具备独立的 `HERMES_HOME`
- Profile 提供独立的配置、记忆、技能、会话和运行环境
- 页面负责管理组织关系与人员信息
- Profile 负责承载实例隔离
- 主 Agent 是默认对话入口，子 Agent 是公司架构中的 Profile Agent 执行单元

### 10.2 页面应展示的 Profile 信息

建议在 Agent 详情中展示：

- 生成的 Profile Agent 名称
- Profile Agent 状态
- Profile Agent 模板来源或生成策略
- 最近同步时间

### 10.3 当前实现边界

当前 Hermes Profile 的底层机制仍以 CLI 侧实现为基础，但 Electron 已通过转接层接入这套能力，因此组织架构页面不仅管理组织关系，也可以驱动底层 Profile Agent 的创建与状态同步。

页面文案中应明确强调：

- 页面管理的是组织架构和 Agent 编排
- Agent 的底层隔离依赖 Hermes Profile
- Electron 页面层通过转接机制调用底层 CLI Profile 能力
- 用户在页面中操作的是 Agent 人员实体，系统在底层自动维护对应的 Profile Agent 运行实例

### 10.4 参考 Hermes Profile 的实现约束

根据 Hermes Profile 文档，Profile 的本质是一个独立目录，内部包含独立的：

- `config.yaml`
- `.env`
- `SOUL.md`
- memories
- sessions
- skills
- Gateway 运行状态与相关文件

因此本页面不应把 Profile 仅理解为一个配置 ID，而应将其视为：

- 一个由系统自动创建的独立 Agent 运行环境
- 一个从组织架构页面派生出来的运行时实例
- 一个与 Agent 人员实体一一对应的隔离工作单元

## 11. 权限与隐私边界

该页面必须遵循企业组织架构文档中的双空间原则。

### 11.1 页面可管理范围

页面可管理内容：

- 公司结构
- 部门结构
- 岗位组织归属
- Agent 组织归属
- Agent 企业角色
- 公司空间相关配置

### 11.2 页面不可触达范围

页面不可提供：

- 个人空间浏览入口
- 个人对话内容查看
- 个人记忆查看
- 个人私有文件访问
- 个人空间导出入口

### 11.3 页面中的说明要求

当用户查看 Agent 详情时，应明确展示：

- 公司空间：企业协作内容，可由企业管理
- 个人空间：仅 Agent 本人可见，本页面不可访问

该提示既是权限说明，也是产品信任边界。

## 13. 存储设计总原则

本页面涉及的数据应拆分为三类存储：

1. SQLite 中的组织主数据
2. 文件系统中的工作目录与运行目录
3. 直接存放在 Profile Agent 系统资源中的 `SOUL.md` / Profile Agent 配置

整体原则如下：

- 组织关系、权限状态、页面展示所需字段，必须以 SQLite 为唯一真源
- 公司、部门、岗位、Agent 各自拥有独立工作目录，工作目录属于文件系统资源
- Agent 的人格、工作风格、行为约束不单独入库，而是直接写入对应 Profile Agent 的系统资源
- Profile Agent 是基于 Agent 自动生成的派生运行对象，不是组织主数据源
- 不允许将组织关系仅写入 `SOUL.md` 而不落库

推荐在 Hermes 现有 `state.db` 之外，单独维护一个组织数据库：

- Electron 根目录下的 `state.db`：Hermes 现有会话与运行时状态
- `org/org.db`：组织架构、人物实体、岗位、目录、Profile Agent 元数据

这样可以避免业务主数据和 Hermes 运行时数据耦合。

在 Electron 模式下，推荐采用以下标准布局：

```text
~/Library/Application Support/hermes-agent-electron/
├── state.db
├── config.yaml
├── SOUL.md
├── org/
│   ├── org.db
│   ├── migrations/
│   ├── workspaces/
│   ├── profiles/
│   └── rendered/
```

其中：

- `state.db` 保持为 Hermes 现有主运行时数据库
- `org/org.db` 专门存储组织编排相关数据
- `org/` 下的其余目录用于承载组织模块的迁移、目录、Profile Agent 和渲染结果

## 14. SQLite 数据模型设计

### 14.1 建议表清单

V1 建议至少包含以下数据表：

- `companies`
- `departments`
- `positions`
- `agents`
- `profile_agents`
- `master_agent_assets`
- `workspaces`

### 14.2 companies

存储公司主信息。

建议字段：

- `id`：自增主键
- `code`：公司稳定编码
- `name`：公司名称
- `goal`：公司一句话目标
- `description`：公司描述
- `status`：状态
- `workspace_path`：公司工作目录
- `created_at`
- `updated_at`

### 14.3 departments

存储部门主信息。

建议字段：

- `id`
- `company_id`
- `parent_id`：可选，支持子部门
- `code`
- `name`
- `goal`
- `description`
- `leader_agent_id`
- `workspace_path`
- `sort_order`
- `status`

- `source_path` 指向主 Agent 当前实际资源位置
- `source_format` 用于描述资源格式，便于后续复制或解析
- `visibility` 决定是否允许被子 Agent 继承
- `inherit_mode` 决定子 Agent 继承时的处理方式

- `copy_to_workspace`：复制到子 Agent 自己的工作空间
- `copy_to_profile`：复制到子 Agent 的 Profile Agent 目录
- `merge_config`：作为配置片段合并进子 Agent 的 Profile 配置
- `inject_prompt`：作为提示词片段注入生成的 `SOUL.md`

- `id`
- `scope_type`：`system / company / department / position / template`
- `scope_id`
- `template_key`
- `requirement_type`：`asset / sql_field / profile_file / env_key / config_key`
- `requirement_key`
- `required_level`：`required / optional`
- `expected_source`：`profile_template / master_agent_assets / sqlite`
- `validation_status`：`ready / missing / blocked`
- `last_validated_at`
- `validation_message`
- `created_at`
- `updated_at`

这张表的作用是：

- 记录某个子 Agent 一键生成时到底需要哪些条件
- 记录这些条件当前是否已经满足
- 在页面上展示“可一键生成 / 缺少必需项 / 存在风险”

典型必需项包括：

- 可用的 Profile 模板

- 岗位职责和岗位目标
- 子 Agent 对应的工作目录
- 可用的基础 skill / tool 资产

### 14.8.1 `config.yaml` 与 `.env` 的继承规则

主 Agent 的配置类资产需要进一步区分：

- `config.yaml`：允许白名单式继承
- `.env`：不建议整文件继承，但允许将运行所必需的 API Key 和白名单变量下发给子 Agent

本节的核心原则是：

**主 Agent 已经配置过且属于白名单的内容，子 Agent 创建时应自动继承，不需要重新配置。**

#### `config.yaml` 可以继承的内容

建议允许子 Agent 继承的 `config.yaml` 内容包括：

- 默认模型配置
- 推理强度、温度等行为参数
- 默认技能开关
- 默认工具开关
- 工作目录策略
- 输出格式偏好
- 非敏感功能开关

这些内容建议通过：

- `asset_type = config`
- `source_format = yaml`
- `inherit_mode = merge_config`

的方式合并到子 Agent 的 Profile 配置中。

#### 当前配置下 `config.yaml` 的明确白名单

结合当前配置文件结构，建议以下字段直接纳入白名单：

- `model.default`
- `model.provider`
- `model.base_url`
- `provider_routing.*`
- `smart_model_routing.*`
- `terminal.backend`
- `terminal.timeout`
- `terminal.lifetime_seconds`
- `terminal.container_cpu`
- `terminal.container_memory`
- `terminal.container_disk`
- `terminal.container_persistent`
- `terminal.docker_image`
- `terminal.singularity_image`
- `terminal.modal_image`
- `terminal.daytona_image`
- `security.tirith_enabled`
- `security.tirith_path`

这些字段都直接影响子 Agent 是否能“创建后可用”，因此不应要求重复配置。

#### `config.yaml` 不可以继承的内容

以下内容不应从主 Agent 直接继承到子 Agent：

- 认证相关配置
- 平台 token 绑定
- OAuth 凭据
- 端口占用配置
- 当前会话上下文
- 与主 Agent 强绑定的本地绝对路径
- 仅适用于主 Agent 的运行时状态

以下字段不建议直接复制原值，而应由系统在创建子 Agent 时重新派生：

- `terminal.cwd`
- 子 Agent 的 workspace 路径
- 子 Agent 的 profile 目录路径

这些内容即使存在于 `config.yaml` 中，也必须在子 Agent 创建时被过滤。

#### `.env` 的默认规则

`.env` 不应整文件原样复制给子 Agent，但子 Agent 要正常运行时，确实需要继承一部分运行必需的环境变量。

因此推荐规则是：

- 不继承完整 `.env`
- 仅继承经过筛选的白名单变量
- 运行模型所需的 API Key 允许继承
- 被公开继承的 tool / skill 所需 API Key 允许继承
- 高风险凭据仍默认禁止继承

#### `.env` 可以继承的内容

只有满足以下条件的环境变量才允许被子 Agent 继承：

- 明确登记到 `master_agent_assets`
- `visibility = public`
- 是运行模型所必需，或被判定为允许下发的配置项
- 有明确目标用途

这类变量通常是：

- 模型提供商 API Key
- 模型提供商 Base URL
- 被公开继承的 tool / skill 所需 API Key
- 非敏感功能开关
- 非敏感默认行为变量
- 非敏感路径类变量

建议这类变量使用：

- `asset_type = config`
- `source_format = yaml`、`json` 或经过筛选后的 env 片段
- `inherit_mode = merge_config` 或 `copy_to_profile`

典型允许继承的变量包括：

- `OPENROUTER_API_KEY`
- `ARK_API_KEY`
- `ARK_BASE_URL`
- `GOOGLE_API_KEY`
- `GEMINI_API_KEY`
- `GEMINI_BASE_URL`
- `OLLAMA_API_KEY`
- `OLLAMA_BASE_URL`
- `GLM_API_KEY`
- `GLM_BASE_URL`
- `KIMI_API_KEY`
- `KIMI_BASE_URL`
- `KIMI_CN_API_KEY`
- `ARCEEAI_API_KEY`
- `ARCEE_BASE_URL`
- `MINIMAX_API_KEY`
- `MINIMAX_BASE_URL`
- `MINIMAX_CN_API_KEY`
- `MINIMAX_CN_BASE_URL`
- `OPENCODE_ZEN_API_KEY`
- `OPENCODE_ZEN_BASE_URL`
- `OPENCODE_GO_API_KEY`
- `OPENCODE_GO_BASE_URL`
- `HF_TOKEN`
- `XIAOMI_API_KEY`
- `XIAOMI_BASE_URL`
- `EXA_API_KEY`
- `PARALLEL_API_KEY`
- `FIRECRAWL_API_KEY`
- `FAL_KEY`
- `BROWSERBASE_API_KEY`
- `BROWSERBASE_PROJECT_ID`
- `BROWSERBASE_PROXIES`
- `TERMINAL_ENV`
- `HERMES_DOCKER_BINARY`
- `TERMINAL_DOCKER_IMAGE`

这些变量之所以允许继承，是因为如果子 Agent 没有这些运行能力相关配置，就无法做到“创建后立即可用”。

但推荐做法仍然是：

- 按白名单筛选变量
- 只下发子 Agent 实际需要的键
- 复制为子 Agent 自己的 Profile 环境配置

而不是直接原样复制主 Agent 的整份 `.env`。

#### `.env` 不可以继承的内容

以下内容默认一律禁止继承：

- Access Tokens
- Refresh Tokens
- OAuth 凭据
- Cookie
- 私有 webhook 密钥
- 私有平台账号绑定信息
- 任何 Secret / Password / Credential
- `SUDO_PASSWORD`
- `TERMINAL_CWD`
- `TERMINAL_SSH_KEY`
- 主 Agent 当前会话绑定变量
- 主 Agent 专属登录态或用户身份变量

补充说明：

- 模型推理所必需的 Provider API Key 例外，可以白名单方式继承
- 被公开继承的 tool / skill 所需 API Key 也可以白名单方式继承
- 但平台账号类、用户身份类、会话类、OAuth 类凭据仍然禁止继承

一句话规则：

- `config.yaml` 采用白名单字段继承
- `.env` 不继承整文件，只允许白名单变量下发
- 运行子 Agent 所必需的 Provider API Key 可以继承
- 被公开继承的 tool / skill 所需 API Key 可以继承
- 主 Agent 已配置好的白名单项，子 Agent 创建时不需要重新配置
- 账号身份类和高风险凭据仍然禁止继承

岗位工作目录适合存放：

- 岗位 SOP

## 16. SOUL 与 Profile Agent 生成规则

### 16.0 主 Agent 与子 Agent 模型

本系统采用以下模型：

- 主 Agent：默认新建对话时使用的默认 Agent
- 子 Agent：公司架构页面中创建和管理的各个 Profile Agent

其中：

- 主 Agent 不属于公司架构中的某个岗位节点
- 子 Agent 才是公司、部门、岗位体系中的实际管理对象
- 公司架构页面的核心作用，是用企业组织结构去管理这些子 Agent
- 子 Agent 的创建、归属、启停和 Profile 状态都应在页面中清晰可见

### 16.1 生成链路

创建 Agent 后，系统应按以下顺序处理：

1. 写入 `agents`
2. 创建 Agent 工作目录
3. 创建 `profile_agents`
4. 校验该岗位 / 模板是否满足一键生成子 Agent 的必需项
5. 初始化 Profile Agent 目录
6. 收集创建 Profile 所需的三类来源数据
7. 基于三类来源数据生成 `SOUL.md`
8. 将人格和行为约束直接写入对应 Profile Agent 资源
9. 回写 `profile_status`

如果第 4 步校验失败，应明确记录失败原因，而不是生成一个不可用的子 Agent。

### 16.2 子 Agent 创建 Profile 的三类来源数据

子 Agent 在创建 Profile Agent 时，应参考以下三类数据：

1. Profile 模板
2. 主 Agent 中可公开继承的数据 / 工具
3. 来自 SQLite 的真实组织数据

#### 来源 1：Profile 模板

模板负责提供岗位或组织层面的基础骨架，例如：

- 默认身份描述
- 默认职责结构
- 默认工作风格
- 默认输出约束
- 默认技能集

#### 来源 2：主 Agent 中可公开继承的资产

子 Agent 创建 Profile 时，可以继承主 Agent 中被标记为公开的能力，例如：

- 公开工具集
- 公开 skill 集
- 公开资源文件
- 公开配置片段
- 公开方法论
- 公开工作规则
- 可复用提示词片段

其中资产继承规则应为：

- 主 Agent 的 skill、tool、resource、config 必须先登记到 `master_agent_assets`
- 只有 `visibility = public` 的记录才允许被子 Agent 继承
- 继承时根据 `inherit_mode` 决定是复制、合并还是注入
- `copy_to_workspace` 类型复制到子 Agent 自己的工作空间
- `copy_to_profile` 类型复制到子 Agent 自己的 Profile Agent 目录
- `merge_config` 类型作为配置片段写入或合并到子 Agent 的配置
- `inject_prompt` 类型作为提示词片段注入生成的 `SOUL.md`
- `config.yaml` 只允许白名单字段继承
- `.env` 不允许整文件复制，只允许白名单变量下发
- 子 Agent 运行必需的 Provider API Key 允许继承到自己的 Profile 环境配置
- 被公开继承的 tool / skill 所需 API Key 允许继承到自己的 Profile 环境配置
- `visibility = private` 的资产不允许被子 Agent 看到或继承

这里强调的是“公开继承”，而不是复制主 Agent 的完整上下文。

### 16.2.1 一键生成子 Agent 的最小可用条件

为了满足“主 Agent 创建后，尽量支持一键生成可用的子 Agent”这一目标，建议将以下项目标记为最小可用条件：

- 存在可用的 Profile 模板
- 存在至少一个可继承的 Provider API Key
- 岗位、部门、公司数据完整
- 子 Agent 工作目录可创建
- Profile Agent 目录可创建
- 至少存在一组基础可继承资产（skill / tool / resource / config）

如果这些条件满足，页面应展示：

- `可一键生成`

如果部分条件缺失，页面应展示：

- `缺少必需项`
- 缺失项列表
- 修复建议

#### 来源 3：来自 SQLite 的真实组织数据

子 Agent 创建 Profile 时，必须注入数据库中的真实组织信息，例如：

- 公司名称与公司目标
- 部门名称与部门目标
- 岗位名称与岗位职责
- Agent 姓名、角色、归属岗位
- 工作目录路径

SQLite 中的数据是结构真源，优先级高于说明性文本。

### 16.3 子 Agent 的运行时约束

子 Agent 在创建 Profile 后，可以在后续继续修改自己的 Profile 资源，但进入实际工作态后，应遵循以下规则：

- 子 Agent 工作时只参考自己的 Profile Agent
- 子 Agent 不直接读取其他子 Agent 的 Profile
- 子 Agent 不直接依赖主 Agent 的实时上下文
- 主 Agent 提供的是创建时可继承的公开能力，而不是运行时共享上下文
- 子 Agent 使用的 skill / tool / resource / config 以落入自己工作空间或自己 Profile 中的副本为准

也就是说：

- 创建时可以参考三类来源
- 工作时只依赖自己的 Profile

### 16.4 什么内容写入 Profile Agent / SOUL

建议直接写入对应 Profile Agent 的 `SOUL.md` 的内容：

- Agent 名称
- 公司名称
- 部门名称
- 岗位名称
- 公司目标
- 部门目标
- 岗位目标
- 岗位职责
- 工作风格
- 输出偏好
- 协作规则
- 禁止事项

这些内容不要求单独落到 SQLite，而是直接保存在对应 Profile Agent 资源目录中。

### 16.5 什么内容不写入 SOUL 作为真源

以下字段即使在 `SOUL.md` 中出现，也只能是说明性文本，不能作为系统真源：

- 数据库主键和外键
- 组织归属关系
- 上级关系
- 头像路径
- 状态字段
- 权限字段
- 工作目录路径
- Profile Agent 路径

### 16.6 SOUL 模板结构建议

```md
# Identity
你是 {{agent_name}}，属于 {{company_name}} / {{department_name}} / {{position_name}}。

# Mission
公司目标：{{company_goal}}
部门目标：{{department_goal}}
岗位目标：{{position_goal}}

# Responsibilities
{{position_responsibilities}}

# Working Style
{{working_style}}

# Output Preferences
{{output_preferences}}

# Hard Rules
{{hard_rules}}
```

### 16.7 Profile 模板设计

为了让不同岗位创建出来的 Agent 具备一致的基础能力，建议在组织架构模块中引入 `Profile Template` 概念。

Profile 模板的作用：

- 定义某一类岗位默认使用的 Profile Agent 初始化规则
- 统一 `SOUL.md` 的基础骨架
- 统一默认技能集、默认记忆策略和默认工作方式
- 减少每次创建 Agent 时重复填写配置

建议关系如下：

- 公司可以定义全局默认模板
- 部门可以定义部门默认模板
- 岗位可以指定岗位默认模板
- 创建 Agent 时优先继承岗位模板，再叠加该 Agent 在 Profile 资源中的个性化内容

推荐优先级：

`Agent 个性化配置 > 岗位模板 > 部门模板 > 公司默认模板 > 系统默认模板`

### 16.8 Profile 模板建议字段

Profile 模板建议优先以文件形式保存在：

- `~/Library/Application Support/hermes-agent-electron/org/rendered/templates/`

模板元信息可包含：

- `id`
- `template_key`
- `name`
- `scope_type`：`system / company / department / position`
- `scope_id`
- `base_soul_md`
- `default_skills`
- `default_memory_policy`
- `default_working_style`
- `default_output_preferences`
- `default_hard_rules`
- `status`
- `created_at`
- `updated_at`

### 16.9 Profile 模板示例

下面是一份适合岗位默认配置的 Profile 模板示例：

```md
# Profile Template: market-analyst-default

# Identity
你是 {{agent_name}}，是 {{company_name}} 的一名正式 Agent 员工，
隶属于 {{department_name}}，当前岗位是 {{position_name}}。

# Organization Context
公司目标：{{company_goal}}
部门目标：{{department_goal}}
岗位目标：{{position_goal}}

# Responsibilities
你的核心职责包括：
{{position_responsibilities}}

# Working Style
- 先澄清任务目标，再开始执行
- 输出结构化、可复用、可追踪
- 优先引用已有资料和工作目录内容
- 对不确定内容明确标注风险和假设

# Collaboration Rules
- 与同部门 Agent 协作时，优先遵守部门目标
- 不越权访问其他 Agent 的个人私有空间
- 公司级输出应进入公司或部门工作目录

# Output Preferences
- 优先使用分点、表格、摘要
- 对分析类任务先给结论，再给依据
- 对执行类任务给出结果、状态和下一步建议

# Hard Rules
- 不泄露个人隐私空间内容
- 不伪造数据、文件或执行结果
- 当权限不足时必须明确说明限制
```

### 16.10 模板与 Agent 的组合方式

创建 Agent 时，推荐按以下顺序生成最终 Profile Agent：

1. 读取岗位关联的 `template_key`
2. 校验 `subagent_bootstrap_requirements` 中的必需项是否满足
3. 加载对应 Profile 模板
4. 从 `master_agent_assets` 中筛选 `visibility = public` 的可继承资产
5. 按 `inherit_mode` 将资产复制、合并或注入到子 Agent 自己的工作空间或 Profile
6. 注入公司、部门、岗位、Agent 的结构化字段
7. 叠加该 Agent 在 Profile 资源中的个性化配置
8. 生成最终 `SOUL.md`
9. 初始化 Profile Agent 目录与基础文件

最终结果应是：

- 模板负责“共性”
- 主 Agent 的公开资产负责“可继承能力”
- Profile Agent 资源负责“个性”
- 组织表负责“结构真相”
- Bootstrap 校验负责“创建后可立即使用”

### 17.1 分层建议

## 21. V1 页面验收标准

V1 页面完成后，至少应满足以下要求：

1. 用户可以创建一个公司，并填写一句话核心目标
2. 用户可以在公司下创建多个部门，并为每个部门填写一句话目标
3. 用户可以在部门下新增、编辑和管理多个岗位
4. 用户可以在岗位下新增、编辑和管理多个 Agent
5. 用户可以清晰查看 `公司 -> 部门 -> 岗位 -> Agent` 的结构关系
6. 用户可以看到 Agent 自动生成的 Profile Agent 信息
7. 页面不提供个人空间访问入口
8. 页面整体逻辑体现“目标驱动的组织编排”

## 23. 总结

本页面的核心不是做一个静态组织树，而是建立一个企业多 Agent 的编排入口。

页面的核心逻辑应始终保持为：

**先定义公司目标，再拆分部门目标，再建立岗位职责，最后把 Agent 编排进岗位中执行目标。**

在这个模型中：

- 公司负责方向
- 部门负责承接目标
- 岗位负责沉淀职责
- Agent 作为人负责执行职责
- Hermes Profile 负责底层隔离

这就是该页面在产品层和架构层上的统一表达。
